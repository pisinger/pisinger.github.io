## ⛵ AKS

| Indicator | Feature | Type | Description |
|---|---|---|---|
| 🔴 | CVE-2026-31431 (Copy Fail) mitigation | Security | Linux kernel `algif_aead` local privilege escalation allows any pod (even non-root) to escalate to root. Mitigation deployed in node images 202604.13.0 and 202604.24.0; upgrade node images immediately. |
| 🔴 | Container Insights nodes/proxy RCE (AKS-2026-0004) | Security | On AKS versions earlier than v1.33, the Container Insights add-on requires `nodes/proxy` GET, which can be abused for RCE via the Kubelet API on port 10250. Upgrade to v1.33+ where the narrower `nodes/pods` subresource is used. |
| 🔴 | TLS 1.2 EMS enforcement (AKS v1.34) | Security | Go 1.25 FIPS mode rejects TLS 1.2 without Extended Master Secret. Applications built with Go older than 1.21 may fail TLS handshakes to FIPS-enabled AKS components after upgrading to v1.34. Rebuild with Go 1.21+. |
| ⚫ | Ingress NGINX retirement | Deprecation | The Kubernetes SIG Network announced retirement of the Ingress NGINX project, with maintenance ending March 2026. Production workloads supported through November 2026. Migrate to Gateway API-based application routing. |
| ⚫ | AKS LTS v1.29 deprecated | Deprecation | AKS Kubernetes Long Term Support version 1.29 is deprecated. Upgrade clusters to a currently supported version. |
| ⚫ | AKS v1.32 LTS-only | Deprecation | AKS Kubernetes version 1.32 is now available only through Long Term Support. Use an LTS plan or upgrade to a standard-supported version. |
| 🟢 | Gateway API-based ingress for app routing | GA | Gateway API-based ingress for the application routing add-on is now generally available as the recommended replacement for Ingress NGINX. |
| 🟡 | NAT Gateway V2 outbound type | Preview | Preview support for AKS-managed NAT Gateway V2 outbound type in supported public Azure regions. |
| 🟡 | AKS-managed GPU metrics | Preview | AKS-managed GPU metrics are now supported by default in Azure Managed Prometheus and Grafana in Azure Monitor. |
| 🟡 | Capacity Based Surge | Preview | Customers can now set both MaxUnavailable and MaxSurge values to surge during node pool upgrades based on available capacity. |
| 🟢 | AKS Automatic ClusterRoleBinding protection | GA | Added protection that blocks `ClusterRoleBinding` create or update requests when the `roleRef` targets configured privileged `ClusterRole`s, reducing privilege escalation risk. |
| 🔵 | Component and image updates | New/Updated | April release updates Azure Policy add-on to 1.16.1, Istio revisions, CSI drivers, Cilium to v1.17.10, Karpenter v1.10.2, cloud-provider-azure, and new Ubuntu/AzureLinux/Windows node images. |
| 🔵 | kube-proxy reduced privileges | New/Updated | Starting with Kubernetes 1.30+, `kube-proxy` uses `NET_ADMIN` and `SYS_RESOURCE` capabilities instead of `privileged: true`. |
| 🔵 | AKS Automatic Gateway API default | New/Updated | Starting on AKS 1.36, new AKS Automatic clusters will be preconfigured with Kubernetes Gateway API via the application routing add-on instead of Managed NGINX ingress. |

## 🏛️ Security Architect Observations

- **AKS security posture requires urgent attention**: two active kernel-level vulnerabilities (CVE-2026-31431 and the Container Insights `nodes/proxy` RCE) demand immediate node image upgrades and version migration planning to v1.33+, including defense-in-depth network restrictions on Kubelet port 10250.
- **The deprecation of Ingress NGINX and AKS LTS v1.29 creates a migration window** that security architects should incorporate into 2026 roadmap planning, prioritizing Gateway API adoption and standard-support Kubernetes versions to avoid unsupported exposure.
- **Defender for Cloud's unified portal integration and GitHub Advanced Security bidirectional sync** strengthens the shift-left paradigm by connecting runtime risk signals with code-level vulnerability management, enabling security-by-design in CI/CD pipelines.
- **Entra ID's cross-tenant group synchronization and passkey-first authentication expansion** signal a move toward decentralized, phishing-resistant identity architecture—architects should evaluate Zero Trust alignment and least-privilege governance across multi-tenant environments.
- **The convergence of Microsoft Foundry agent governance, ACA Dynamic Sessions, and Purview Agent 365 protections** introduces new trust boundaries for AI workloads requiring data residency, sandboxing controls, and observability across agent lifecycles.
- **New Defender for Identity alerts for stolen session cookies and Conditional Access bypass** highlight the need for hybrid identity monitoring that spans on-premises Active Directory and cloud Entra ID in unified detection logic and incident response architecture.

---

## 🛠️ Security Operations Observations

- **SOC teams must prioritize AKS node image upgrades** due to confirmed non-root pod-to-root exploits; existing nodes created before hotfix VHDs remain vulnerable and require immediate self-service mitigation or node pool replacement.
- **Defender for Identity's eight new Entra ID-focused alerts** (guest elevation, Global Admin creation, stolen session replay, suspicious MFA addition) require immediate tuning of incident response playbooks, severity mappings, and escalation procedures.
- **Sentinel's AI playbook generator and UEBA Okta V2/GCP expansion** reduce detection engineering backlog but require validation of generated automation logic and anomaly thresholds to prevent false positives and alert fatigue.
- **Automatic device isolation in Defender XDR/Endpoint preview changes SOC containment workflows**—operators should establish clear release criteria, monitor for over-isolation of production systems, and document exception paths for business-critical assets.
- **Defender for Cloud's daily secure score snapshot recalculation** may cause score trend anomalies in existing dashboards; SOC reporting should be updated to reflect end-of-day values rather than intra-day averages to avoid misleading posture trends.
- **The standalone Defender Experts for Servers offering** provides an alternative to building in-house 24/7 server monitoring; operations teams should evaluate SLA coverage gaps across AWS, GCP, and on-premises estates against existing threat detection capabilities.
