# Microsoft Security Release Radar - June 2026

## 🛡️ Microsoft Defender Cloud

| Indicator | Feature | Type | Description |
|---|---|---|---|
| 🟢 | Defender for Open-Source Relational Databases on AWS RDS | GA | Database threat protection and sensitive data discovery for Aurora PostgreSQL, Aurora MySQL, PostgreSQL, MySQL, and MariaDB on AWS RDS is now generally available. Previously preview instances begin billing starting July 2026. |
| 🔴 | Container-level misconfiguration recommendations for Kubernetes | Preview | Agentless KSPM now assesses individual containers rather than entire clusters for misconfigurations such as privilege escalation, read-only root filesystem, trusted registries, and automounting API credentials. ## Security Architect Observations

- **Container security granularity shift:** The migration from cluster-level to container-level KSPM in Defender for Cloud significantly improves defense-in-depth but requires architecture teams to redesign remediation workflows and RBAC mappings to handle higher-fidelity findings.
- **Multi-cloud database protection maturation:** GA of AWS RDS open-source database protection signals continued investment in cross-cloud security posture, reducing the need for separate third-party database activity monitoring tools in hybrid AWS/Azure environments.
- **Serverless coverage expansion:** With serverless protection now GA, security architects should update asset inventory and threat models to include Azure Functions, Web Apps, and AWS Lambda as first-class protected resources.
- **Deprecation risk:** Five cluster-level recommendations (HostPath, allowed ports, host networking/ports, CAP_SYS_ADMIN, and AppArmor) are scheduled for deprecation. Teams relying on these for compliance dashboards or automated remediation must transition before GA.
- **AI agent governance emerging:** Foundry’s agent governance and observability capabilities represent an early signal that organizations should begin extending identity, access, and audit controls to AI-hosted agents, not just traditional compute.

---

## Security Operations Observations

- **Alert volume tuning required:** Container-level KSPM will generate more granular findings than the previous cluster-level assessments. SOC teams should prepare for increased initial alert volume and tune severity thresholds to avoid analyst fatigue.
- **Improved AKS remediation MTTR:** The new actionable AKS upgrade recommendation replaces non-actionable guidance, which should reduce mean time to remediate system pod vulnerabilities by providing direct, resolvable remediation paths.
- **Agentless operational benefit:** The new KSPM recommendations use an agentless architecture, eliminating sensor deployment and maintenance overhead on Kubernetes nodes while maintaining continuous posture assessment.
- **Serverless monitoring gaps closing:** SOC visibility into serverless runtime environments has historically been limited; the GA serverless protection capability provides new detection and misconfiguration data streams that should be integrated into SIEM and SOAR playbooks.
- **False positive review for container findings:** Transitioning to container-level findings may surface previously hidden but low-risk configurations (e.g., dev containers running as root). Operations teams should establish a baseline review cycle for the first 30 days of the preview.
